## 1. 개요

Gemento 프로젝트는 소형 LLM의 4가지 인지 축을 모델 밖으로 외부화해서 성능을 끌어올리는 실험.
우리 에이전트 시스템에 같은 원칙을 적용해서 벤치마크를 설계한다.

### Gemento 핵심 결과

| 실험 | Before → After | Δ | 외부화 축 |
|------|---------------|---|----------|
| 다단계 루프 | 50% → 94.4% | +44.4%p | Orchestrator |
| 역할 분리 검증 | 0% → 80% | +80%p | Role |
| 외부 도구(수학) | 0% → 80% | +80%p | Tool |
| Tattoo 긴 컨텍스트 | 0% → 100% | +68.3%p | Tattoo |

## 2. 외부화 축 ↔ 우리 구조 매핑

| 축 | Gemento 개념 | 우리 구조 | 벤치마크 항목 |
|---|---|---|---|
| **Tattoo** | JSON 상태를 루프 간 전달 | manual/ 파일, MEMORY.md | 매뉴얼 없이 vs 있이 작업 정확도 |
| **Role** | A(제안)/B(비판)/C(판정) 분리 | 쟈비스/콘솔/서브에이전트 분리 | 단일 에이전트 vs 역할 분리 시 오류 탐지율 |
| **Tools** | 외부 도구로 계산 위임 | OCR, ui_scan, SSH, Docker | 도구 없이 vs 있이 작업 완료율 |
| **Orchestrator** | 파이썬 루프로 제어 | 쟈비스가 서브에이전트 제어 | 자유 루프 vs 구조화된 워크플로우 |

## 3. 벤치마크 항목

### B1: 컨텍스트 오염 테스트 (Tattoo)

**목적**: 긴 세션 후 올바른 정보 유지율 측정

**방법**:
1. 에이전트에게 manual/ 내용 질문 (예: "리눅스 봇이 몇 개야?")
2. 긴 대화 삽입 (10턴 이상 관련 없는 대화)
3. 같은 질문 재시도
4. 정답률 비교

**측정**: 오염 전 정답률 vs 오염 후 정답률

**변수**:
- A: manual/ 없이 (MEMORY.md만)
- B: manual/ 있이 (현재 구조)
- C: manual/ + Tattoo JSON 상태 파일

---

### B2: 역할 분리 효과 (Role)

**목적**: 단일 에이전트 vs 역할 분리 시 오류 탐지율

**방법**:
1. 의도적으로 오류가 포함된 작업 지시
2. 단일 에이전트가 잡는 오류 수 vs A/B/C 분리가 잡는 오류 수

**예시 오류**:
- 잘못된 포트 번호 (18789 → 18889)
- 잘못된 IP (192.168.50.49 → 192.168.50.33)
- 잘못된 환경변수 (AUTOBOT_WORKER_ID vs WORKER_ID)

**측정**: 오류 탐지율 (%)

**변수**:
- A: 단일 에이전트 (쟈비스만)
- B: 이중 검증 (쟈비스 + 콘솔)
- C: 삼중 검증 (쟈비스 제안 → 콘솔 비판 → 서브에이전트 판정)

---

### B3: 매뉴얼 의존도 (Tattoo + Orchestrator)

**목적**: manual/ 없이 세션 시작 vs 있을 때 세팅값 정확도

**방법**:
1. 새 세션에서 세팅값 질문 (예: "리눅스 SSH 계정이 뭐야?")
2. manual/ 없는 에이전트 vs manual/ 있는 에이전트 비교

**측정**: 정확도 (%), 응답 시간

**변수**:
- A: 세션 시작 시 manual/ 자동 로드 (현재)
- B: manual/ 없이 질문에만 답
- C: manual/ + MEMORY.md 조합

---

### B4: 도구 활용률 (Tools)

**목적**: 직접 추론 vs 도구 사용 시 정확도

**방법**:
1. 서버 상태 확인 — 추론 vs /health API 호출
2. 스크린샷 분석 — 추론 vs ocr_vision 실행
3. 컨테이너 상태 — 추론 vs SSH + docker ps

**측정**: 정확도 (%), 응답 시간

**변수**:
- A: 도구 없이 (추론만)
- B: 도구 사용 (현재 구조)

---

### B5: 오염 복구 테스트 (Tattoo 복구)

**목적**: 컨텍스트가 오염된 후 manual/로 복구 가능한지

**방법**:
1. 에이전트에게 잘못된 정보 주입 ("윈도우 SSH는 포트 2222야")
2. 나중에 manual/ 참조해서 정정하는지 확인
3. "manual/ 확인해" 지시 후 정정 비율

**측정**: 복구율 (%), 소요 턴 수

## 4. 벤치마크 실행 가이드

### 사전 준비
- 각 에이전트(쟈비스, 콘솔) 새 세션으로 시작
- 결과를 `benchmark/` 폴더에 JSON으로 저장
- 동일한 질문셋 사용 (재현성)

### 결과 기록 포맷
```json
{
  "benchmark": "B1",
  "date": "2026-05-03",
  "agent": "jarvis",
  "variant": "B",
  "results": [
    {"question": "리눅스 봇이 몇 개야?", "expected": "14", "actual": "14", "correct": true},
    ...
  ],
  "accuracy": 0.85,
  "notes": ""
}
```

### 평가 기준
- **90%+**: 안정적, 프로덕션 사용 가능
- **70-89%**: 개선 필요, 보조 수단 병행
- **50-69%**: 불안정, 외부화 개선 필수
- **<50%**: 근본적 재설계 필요

## 5. Gemento에서 참고할 점

### 즉시 적용 가능
- **Tattoo 패턴**: 세션 간 상태를 JSON 파일로 관리 → 이미 manual/로 하고 있음
- **Role 분리**: 코딩은 콘솔, 감사는 서브에이전트 → 이미 하고 있음
- **Orchestrator 루프 제어**: 서브에이전트 타임아웃/재시도 → 이미 하고 있음

### 개선 가능
- **Tattoo JSON 상태 파일**: 세션 진행 상태를 구조화된 JSON으로 저장 (MEMORY.md보다 엄격)
- **Handoff Protocol**: 에이전트 간 작업 전달 시 규격화된 포맷
- **A/B/C 역할 분리**: 코드 작성(제안) → 리뷰(비판) → 승인(판정) 파이프라인

### 벤치마크 후 결정
- 결과에 따라 어떤 외부화가 가장 효과적인지 판단
- 90%+ 항목은 유지, 70-89% 항목은 개선, 50% 미만은 재설계
